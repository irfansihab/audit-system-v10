Menjadikan Audit AI v7 sebagai mesin AI per-tahap di belakang siklus pengawasan SIMWAS — bukan backend umum, bukan aplikasi terpisah.
Inspektorat II, Komdigi · SIMWAS = orkestrator & system of record · v7 = engine analisis (agen + skill) · disusun 27 Mei 2026

1. Ringkasan & Keputusan

SIMWAS v2 (modul INTEGRAL) sudah jadi orkestrator lengkap siklus pengawasan dan model datanya nyaris 1:1 dengan domain v7 (Penugasan → PKP → Kertas Kerja → Temuan & Rekomendasi → LHP). Maka integrasinya = plug-in AI di tahap-tahap tertentu: SIMWAS tetap memegang workflow, data, UI, dan template; v7 dipanggil untuk pekerjaan AI-berat lalu mengembalikan hasil terstruktur. Karena tim SIMWAS akan menyediakan API/SSO/webhook sesuai yang kita minta, dokumen ini sekaligus menjadi kontrak yang v7 ajukan.

1 · REST APIDisediakan tim SIMWAS sesuai permintaan v7. → Dokumen ini mendefinisikan endpoint yang dibutuhkan (bagian 4).
2 · Format hasilKKP: v7 balikkan JSON terstruktur sesuai format KKSA SIMWAS. Draft laporan: v7 balikkan dokumen Word (.docx).
3 · Template KriteriaSaat ini kosong di SIMWAS → digantikan skill + references di v7. v7 jadi sumber kriteria/prosedur pengawasan.
4 · Bukti Cek AITahap "Bukti Cek AI" dihapus dari workflow. Pilot pindah ke tahap Kertas Kerja.
5 · SSOMenyesuaikan permintaan v7. → v7 ajukan JWT bertanda-tangan SIMWAS (JWKS); alternatif Sanctum introspection.
6 · WebhookMenyesuaikan permintaan v7. → v7 ajukan event saat tahap dibuka + tanda tangan HMAC (pola yang sudah dipakai integrasi CACM).

Tiga kanal penghubung:

  • SSO — SIMWAS menerbitkan token identitas; v7 memverifikasi (tanpa login kedua).
  • v7 → SIMWAS (REST, tarik) — v7 mengambil Penugasan, PKP (sasaran+langkah), dan dokumen yang di-upload.
  • SIMWAS → v7 (webhook/REST, picu) & v7 → SIMWAS (REST, tulis-balik) — SIMWAS memicu analisis; v7 menulis Temuan & Rekomendasi (JSON) / draft LHP (.docx) kembali ke SIMWAS.

Pemetaan peran (identik): Penanggung Jawab → Pengendali Mutu (PM) → Pengendali Teknis (PT) → Ketua Tim (KT) → Anggota Tim (AT). v7 sudah punya PT/KT/AT/PM; cukup tambah PJ.

4. Kontrak integrasi v7 mengajukan

4.1 SSO — identitas dari SIMWAS

v7 mengusulkan JWT bertanda-tangan SIMWAS (RS256), diverifikasi v7 via endpoint JWKS publik SIMWAS — stateless, tak perlu round-trip tiap request. Token dikirim sebagai Authorization: Bearer <jwt> (atau saat embed UI v7, diteruskan SIMWAS via query/cookie sekali lalu ditukar sesi v7).

// Klaim JWT minimum yang v7 butuhkan
{
  "sub": "1042",                 // user id SIMWAS
  "name": "Sarah Rahmawati",
  "role": "AT",                // PJ | PM | PT | KT | AT
  "penugasan_id": "255",       // konteks penugasan aktif (opsional)
  "iss": "simwasv2.komdigi.go.id",
  "exp": 1748370000
}

Alternatif bila SIMWAS lebih mudah: token Laravel Sanctum + endpoint introspeksi GET /api/me (v7 panggil sekali untuk validasi + ambil identitas). JWKS lebih disukai karena tanpa beban round-trip.

Pola async (job_id + status) dipakai karena run agen LLM butuh waktu > batas timeout HTTP biasa. Bisa juga v7 push hasil balik via 4.2 saat selesai (callback), supaya SIMWAS tak perlu polling.

4.4 Webhook — SIMWAS memberi tahu v7

SIMWAS memanggil POST /integrasi/webhook saat tahap dibuka, dengan header X-Simwas-Signature (HMAC-SHA256 atas body, secret bersama) — v7 sudah punya verifikasi HMAC dari integrasi CACM, tinggal pakai ulang.

{
  "event": "penugasan.kertas_kerja.started",   // atau .konsep_laporan.requested
  "penugasan_id": "255",
  "anggota_id": "77",
  "ts": "2026-05-27T14:00:00Z"
}

5.3 Temuan & Rekomendasi — JSON KKSA (kontrak inti tahap 3)

Kolom UI SIMWAS: Referensi Sasaran | Judul Temuan | Uraian Rekomendasi (+ detail via Aksi). v7 mengembalikan superset terstruktur; SIMWAS memetakan kolom utama dan menyimpan detail.

{
  "penugasan_kode": "255/IJ.3/KP.01.06/11/2025",
  "kkp_no": "KKP-1/255/IJ.3/KP.01.06/11/2025",
  "disusun_oleh": "Sarah Rahmawati",
  "temuan": [
    {
      "referensi_sasaran": "Memastikan kesesuaian antar dokumen TOR, RAB, dan Matriks…",
      "sasaran_id": "S-1",
      "status": "Belum Sesuai",        // "Sesuai" bila tak ada temuan
      "judul_temuan": "Rincian RAB di luar SBM tanpa dokumen pembentuk harga",
      "kondisi": "…",            // detail
      "kriteria": "Perpres 16/2018 Pasal 26 …",  // detail (sitasi)
      "akibat": "…",             // detail
      "uraian_rekomendasi": "…",
      "severity": "HIGH",
      "dokumen_sumber": [
        { "file": "RAB-PED.pdf", "halaman": 4, "kutipan": "…" }
      ]
    }
  ]
}
Yang perlu dikunci bersama tim SIMWAS: nilai sah status (Sesuai / Belum Sesuai / …), batas panjang teks, dan apakah dokumen_sumber (sitasi halaman — jejak anti-halusinasi v7) disimpan SIMWAS atau hanya ditampilkan. Sitasi ini inti reproducibility v7, sebaiknya dipertahankan.

7. Fase implementasi

FASE 0  Kontrak & fondasi (bareng tim SIMWAS)

  • Sepakati spesifikasi OpenAPI dua arah (bagian 4) + skema JSON KKSA (5.3) + format token SSO (4.1) + event webhook (4.4).
  • Sediakan lingkungan staging SIMWAS + akun uji per role + base URL API.
  • Putuskan kepatuhan data (lihat bagian 8) — gating sebelum data riil mengalir.

FASE 1  Pilot — tahap Kertas Kerja (KKP) E2E

  • SSO (V1) + mirror penugasan (V2) + tarik dokumen (4.2) untuk 1 penugasan reviu/audit nyata di staging.
  • Jalankan agen AT → serializer KKSA (V3) → tulis-balik ke POST …/kkp/{anggota}/temuan.
  • Verifikasi: Temuan & Rekomendasi muncul benar di UI KKP SIMWAS; sitasi dokumen_sumber utuh.

FASE 2  Konsep Laporan + reviu supervisi

  • Tahap 5: KT susun narasi → .docx → POST …/konsep-laporan (V6).
  • Tahap 4 & 6: QC SAIPI → catatan reviu (LRS) ke SIMWAS.
  • Webhook (V5): SIMWAS picu otomatis saat tahap dibuka — kurangi aksi manual.

FASE 3  Kriteria dari v7 + roll-out

  • Endpoint kriteria (V7): SIMWAS tampilkan kriteria/langkah dari skill v7 sebagai pengganti Template Kriteria kosong.
  • (Opsional) embed UI v7 sebagai sub-halaman di SIMWAS (reverse-proxy + SSO) untuk interaksi chat/gate yang kaya.
  • Roll-out ke seluruh jenis pengawasan — karena engine sudah generik (skill folder-driven), tak ada kode per-jenis.

8. Risiko & keputusan terbuka

Kepatuhan data (paling kritis). Walau v7 dipindah on-prem/PDN, inferensi LLM tetap keluar ke Anthropic. Untuk isi dokumen auditee + temuan, perlu keputusan: (a) kirim ke Anthropic dengan perjanjian zero-retention/enterprise, (b) gateway redaksi data sensitif sebelum kirim, atau (c) model on-prem. Ini menentukan kelayakan seluruh rancangan untuk data riil dan harus diputus di Fase 0.
HalCatatan / mitigasi
Deploy v7 di PDNv7 kini di Fly.io (SG) + Vercel → untuk integrasi data riil, pindah on-prem/PDN dekat SIMWAS (sambung item "Migrasi PDN Tahap-2").
Skema KKSA dikunciFormat Temuan & Rekomendasi harus disepakati & di-versioning; perubahan kolom SIMWAS = perubahan kontrak.
Ketergantungan tim SIMWASAPI/SSO/webhook dikerjakan tim lain → Fase 0 (kontrak) jadi jalur kritis; sediakan mock server di v7 agar pilot tak menunggu.
Durasi run agenLLM bisa lama → wajib async (job_id) + callback; jangan blokir request SIMWAS.
Sitasi anti-halusinasiPertahankan dokumen_sumber (file+halaman+kutipan) — pembeda v7; pastikan SIMWAS menampung/menampilkan.
Identitas gandaSetelah SSO, hapus seed user lokal v7 di jalur SIMWAS; hindari dua sumber kebenaran identitas.
